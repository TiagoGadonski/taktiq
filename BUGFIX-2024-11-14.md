# Bug Fixes - November 14, 2024

This document details critical bug fixes applied to the GymHero2 (TaktIQ) application.

## Overview

Two critical issues were identified and resolved:
1. **User Deletion Failure** - Admin accounts unable to delete users with activity logs
2. **Home Workout Generation Failure** - No exercises available for home/at-home workouts

---

## Issue 1: User Deletion Failure

### Problem Description

When attempting to delete a user through the admin panel, the operation would fail with a foreign key constraint violation error if the user had any activity logs.

**Error Code:** `23503`
**Error Message:** `update or delete on table "Users" violates foreign key constraint "FK_UserActivityLogs_Users_UserId" on table "UserActivityLogs"`

### Root Cause

The foreign key constraint between `UserActivityLogs` and `Users` tables did not have an `ON DELETE` behavior specified. By default, PostgreSQL uses `RESTRICT`, which prevents deletion of referenced records.

### Solution

**Migration:** `20251114114209_FixUserActivityLogsForeignKey`

The FK constraint was updated to use `ON DELETE SET NULL` behavior:

```sql
ALTER TABLE "UserActivityLogs"
DROP CONSTRAINT IF EXISTS "FK_UserActivityLogs_Users_UserId";

ALTER TABLE "UserActivityLogs"
ADD CONSTRAINT "FK_UserActivityLogs_Users_UserId"
FOREIGN KEY ("UserId")
REFERENCES "Users" ("Id")
ON DELETE SET NULL;
```

**Code Changes:**

1. **Migration File:** `src/GymHero.Infrastructure/Migrations/20251114114209_FixUserActivityLogsForeignKey.cs`
2. **ApplicationDbContext:** `src/GymHero.Infrastructure/Data/ApplicationDbContext.cs`
   - Added explicit configuration for `UserActivityLog` entity relationship

### Impact

- ✅ Users can now be deleted even if they have activity logs
- ✅ Activity logs are preserved for audit purposes (UserId becomes NULL)
- ✅ No data loss - logs remain in the system for compliance and analytics
- ✅ Orphaned logs can be queried with `WHERE UserId IS NULL`

### Testing

Verified with test script that:
1. Users with activity logs can be deleted successfully
2. Activity logs remain in the database with `UserId = NULL`
3. No constraint violation errors occur

Investigation revealed that **all 119 exercises** in the database were marked as `WorkoutLocation = 0` (Gym Only). There were:
- **0 exercises** marked as Home Only (WorkoutLocation = 1)
- **0 exercises** marked as Both (WorkoutLocation = 2)

The AI workout generation logic had no exercises to select from when filtering for home workouts.

## Update - November 14, 2024 (Evening)

### Additional Fix: AI Still Generating Gym Workouts for Home Users

**Issue:** Even after populating home exercises in the database, the AI was still generating gym workouts with equipment (dumbbells, barbells, etc.) when users requested home workouts.

**Root Cause:** The AI prompt mentioned "equipamento mínimo" (minimal equipment) which the AI interpreted to include dumbbells and other gym equipment.

**Solution:** Strengthened AI prompt enforcement with much more explicit instructions:

1. **Enhanced Location Context (lines 1568-1603):**
   - Changed from "equipamento mínimo" to "ZERO EQUIPAMENTO DE ACADEMIA"
   - Added explicit list of allowed bodyweight exercises
   - Added explicit list of ABSOLUTELY PROHIBITED equipment
   - Added examples of prohibited exercises
   - Multiple layers of warnings with emojis for emphasis

2. **Updated System Prompt (lines 1095-1100):**
   - Changed "APENAS" to "EXCLUSIVAMENTE"
   - Changed "equipamento mínimo" to "PESO CORPORAL"
   - Added "ABSOLUTAMENTE PROIBIDO" language
   - Added validation rule: equipment field MUST be "peso corporal" for home workouts

**File Changed:** `src/GymHero.Api/Endpoints/AIEndpoints.cs`

**New Home Workout Prompt:**
```
🏠🏠🏠 TREINO EM CASA - ZERO EQUIPAMENTO DE ACADEMIA 🏠🏠🏠

✅✅✅ PERMITIDO (PESO CORPORAL APENAS):
• Flexões (todas as variações)
• Agachamentos livres (sem peso)
• Afundos (sem peso)

"máquina", "cabo", "kettlebell", "anilha" ou similar, você FALHOU COMPLETAMENTE.
```

**Impact:**
